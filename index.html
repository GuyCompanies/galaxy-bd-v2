Cascade INACTIVE → Review Queue

The pattern that closes Landmine #1 without hard cascade or migrations

How it works

  • User flips Brand X → INACTIVE.
  • System computes impacted children: 14 Deals, 6 live Campaigns, 23 CM allocations.
  • Modal shows the impact before confirming.
  • User types the Brand name to confirm (type-to-confirm pattern).
  • Brand flag flips. All children's effectiveStatus recomputes via resolver — no FK cascade, no row deletes.
  • Children show CASCADE-INACTIVE badge with link back to parent.
  • Reactivation reverses in 1 click (parent flip → children recompute).

Why it's safe

  • Soft flag, not hard cascade — no risk of FK orphaning.
  • Resolver-computed — single source of truth, no dual-write drift.
  • Audit trail baked in — every flip writes who/when/why.
  • Reversible — flip parent back, children come back.
  • Cron-pause Campaigns — live Campaigns running on cascade-inactive Deals auto-pause within 60s. Postbacks stop. Affiliate manager notified.
  • One pattern reused across Network System / Program / Platform / Brand / Account.
⚠️
Edge case to flag at meeting: what if a CM is mid-campaign when their Account flips INACTIVE? Current proposal: campaign continues to current funnel-flush (~24h), then auto-pauses. Open question for Noa: do we hard-pause within 60s or do a 24h grace window? Default recommendation: 60s pause + automated CM notification.

4 UX Tensions — Resolved

Where team voices disagreed · the call we recommend
#TensionVoicesResolutionWhy
1 Welcome offer scope: per-Deal or per-Country? Pnina: per-Country (varies by market). Yonatan: per-Deal (simpler). Idan: indifferent. Per-Country. Lives inside the DealGeo block. Inherits the Deal default but country can override. Welcome offers vary by promo law (IT vs ES vs PT). Per-Country is the only safe default.
2 Verticals: own page or filter chip? Noa: own page (treat as primary IA). Pnina: filter chip on Programs page. Own page in Segment Management tab. Vertical is a first-class organizational axis, not a filter. Verticals will grow (Casino, Sportsbook, Lottery, Crypto, ...). Filter chips don't scale past 5.
3 "+ Add New" pattern in Create Deal cascade Pnina: modal popup. Noa: sidesheet that doesn't lose form state. Right-side sidesheet. Inline create without losing partially-filled Deal form. Closes back to the cascade with new value pre-selected. Modal loses form state if user closes mid-create. Sidesheet preserves it. Standard AMP pattern.
4 Bank altitude: Program or Account? Pnina: Program-level default. Idan: Account-level (per Fernando IT vs Fernando ES). Program-default-with-Account-override. Bank fields live on Program. Account can override if needed (rare). UI shows inherited vs overridden badge. ~90% of accounts inherit from Program. Override exists for the 10% case (e.g., Fernando ES uses a different IBAN). Defaults reduce data entry.
✅
All 4 resolutions are reversible. If a call turns out wrong in beta, every one of them is a config flag flip or a UI re-route — no schema rollback needed. Low regret cost on every line.

3 Wild Cards

Out-of-the-box options surfaced for Guy's decision · not in the 3-wave plan but worth considering
WILD CARD 1

OrgUnit enum across the kernel

Today: Network System, Program, Brand, Platform, Account are 5 separate entities. Wild card: collapse them under a generic OrgUnit enum + parent_id self-reference. One table, one cascade resolver, infinite nesting.

Win: Adds Vertical, Region, anything else as zero-code extensions.
Cost: Loses type safety in TypeScript. Reports get harder. ~2 extra dev-days now, saves ~10 days over 12 months.
Recommendation: defer to v3 unless we see a 3rd entity type added in next 6 months.
WILD CARD 2

Program Health as monetization

Every Program × Brand × Account combo has live health signals (FTD trend, payout speed, postback failure rate, dispute count). Wild card: expose this as a "Program Health Score" service to networks themselves as a paid API.

Win: Networks pay us to tell them which of their programs are leaking. New revenue line, B2B SaaS pricing.
Cost: Productizing this is ~1 month after Wave 3 ships. Compliance review needed.
Recommendation: park as "interesting" — surface again after Galaxy 3k stable.
WILD CARD 3

Postback URL on Account (not Program)

Already implied by Landmine #4 resolution, but worth surfacing: postback URLs move from Program → ProgramAccount. Fernando's IT account fires to IT-postback.galaxy.com; ES account to ES-postback.galaxy.com.

Win: Cleaner attribution. Lower postback failure rate. Per-Account A/B testing.
Cost: Account-grain naturally absorbs this — ~0 incremental cost if we model ProgramAccount on day one.
Recommendation: ship in Wave 1. Free win.
💡
Why surface wild cards now: the architecture is being reset for Galaxy. This is the cheapest moment in the next 2 years to embed optionality. Decisions deferred now cost 5-10× to retrofit later.
